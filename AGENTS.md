# 猫步下载器开发注意事项与强约束

本文件适用于整个仓库。任何开发者、自动化工具或 AI Agent 在修改代码前都必须完整阅读并遵守。除非项目负责人明确批准，不得绕过标记为“强约束”的规则。

## 1. 产品定位

- 正式中文名是“猫步下载器”，英文名是 “Maobu Fetch”。面向 Windows 10/11，浏览器范围为 Chrome 和 Edge。
- 产品是本地优先、紧凑高效的下载管理器，不是带营销首页的云服务，也不是 AI 概念产品。
- 当前范围不包括 BT、磁力、账号同步、远程下载、云端服务、DRM 绕过、Firefox 和移动端功能。未经明确批准不得自行加入。
- 不得加入广告、遥测、用户跟踪、在线字体、强制登录或与下载无关的后台网络请求。

## 2. 兼容性强约束

- 不得随意修改 Tauri 标识 `app.lumaget.desktop`、数据库名 `lumaget.db`、分片临时后缀 `.lumaget`、本地端口 `17433`、`X-Luma-*` 协议头和已有命令名。这些旧内部标识为升级兼容而保留，不属于未完成的品牌重命名。
- 如确需修改上述标识，必须同时提供自动迁移、向后兼容期、回滚方案和覆盖旧版本升级的测试。
- SQLite 变更必须使用增量迁移。禁止删除数据库、清空用户任务或通过重建表规避迁移。
- 新增序列化字段必须提供安全默认值，保证旧数据库、旧 JSON 和旧扩展请求仍可读取。
- 不得改变已完成任务的目标文件、已有分片或用户设置，除非操作由用户明确触发。

## 3. 下载内核强约束

- “多线程下载”在本项目中必须实现为并发 HTTP Range 连接，不得通过启动多个独立程序进程伪装多线程。
- 每任务连接数只能为 `1 / 2 / 4 / 8 / 16 / 32`，任务级设置优先于全局默认值；新任务必须保持“连接数 = 分片数”，避免额外 Range 轮换导致 CDN 限速。旧版动态分片仅允许按原布局安全续传。
- 仅在服务器支持 Range、资源长度已知且满足分段阈值时启用多连接；否则必须安全回退到单连接。
- 每个 Range 响应必须校验 HTTP `206`、`Content-Range`、起止偏移和最终分片长度。任何分片不完整都不得合并为完成文件。
- 续传必须记录并比较 ETag 与 Last-Modified。远端资源变化时不得继续拼接旧分片。
- 合并必须写入临时文件，完成长度或校验通过后再原子替换目标文件。禁止直接把半成品暴露为完成文件。
- 暂停、取消、重试和程序退出必须停止所有活动连接，并保持可恢复状态。禁止出现重复启动、状态回退、暂停后继续写入或完成后再次下载。
- 全局限速和单任务限速必须覆盖普通流与所有分段连接，不能只限制某一路连接。
- 下载进度、速度、剩余时间和活动连接数必须来自真实状态，不得使用模拟数据。
- 认证信息、Cookie、Authorization、代理密码和持久令牌不得写入日志、错误历史或前端调试输出。

## 4. UI 与品牌强约束

- 视觉必须保持紧凑、克制、专业的 Windows 11 效率型风格：184px 分类导航、约 40px 命令栏、约 36px 表格行高、全宽任务表和可折叠底部详情栏。
- 禁止重新加入玻璃光斑、彩色渐变背景、营销统计卡、大面积留白、卡片堆叠、AI 风装饰或联网 Google 字体。
- 默认跟随系统深浅色；必须同时检查浅色、深色、100%/125% 缩放以及 1280×720、1440×900 窗口。
- 桌面端、安装器和扩展必须统一使用“猫步下载器 / Maobu Fetch”名称与猫形下载图标。
- 图标唯一设计源是 `src-tauri/icons/app-icon.svg`。修改后必须运行 `pnpm tauri icon src-tauri/icons/app-icon.svg`，并同步更新 `extension/src/icon128.png`。
- 图标需在 16–32px 下保持可辨识；禁止使用照片、复杂纹理、文字、小细节或未经授权的第三方素材。
- 生产环境禁止演示任务和伪造统计。加载、空状态、错误、离线、权限不足、磁盘空间不足都必须有明确反馈。
- 交互不能只依赖颜色；多选、键盘、右键菜单、焦点状态和禁用状态必须可识别。

## 5. 浏览器扩展与本地桥强约束

- 本地桥只能监听 `127.0.0.1`，不得监听公网地址。
- 不得开放通配 CORS。必须保留精确扩展 Origin 校验、一次性配对码、持久令牌、HMAC 签名、时间戳校验、速率限制和撤销配对。
- 修改 `/v1` 请求结构、签名算法或请求头时，必须同步更新 Rust、扩展协议代码和协议测试，并说明兼容策略。
- 桌面端离线、未配对或发送失败时，浏览器下载必须安全回退，不能直接取消并丢失用户下载。
- 扩展接管规则必须尊重临时绕过、域名规则、文件类型和大小阈值；不得静默接管所有下载。
- 页面媒体探测仅收集完成当前操作所需的信息，不得上传浏览历史或页面数据。

## 6. 媒体工具与安装包强约束

- yt-dlp、FFmpeg、FFprobe 不得进入基础安装包或程序安装目录。普通 HTTP 下载必须始终开箱即用。
- 媒体工具只能在用户确认后按需安装到应用数据目录，并使用固定版本、可信来源和 SHA-256 清单。
- 安装器必须支持取消、断点、重试、代理、空间检查和失败清理；校验失败时不得切换为可用版本。
- FFmpeg 压缩包只能提取 `ffmpeg.exe` 与 `ffprobe.exe`，必须阻止绝对路径和 `..` 路径穿越。
- 自动更新只能检查并提醒，不得后台自动下载。DRM 内容必须明确拒绝，不得尝试绕过。
- NSIS/MSI 基础安装包必须小于 30 MB。发布前必须检查包内不存在 `yt-dlp.exe`、`ffmpeg.exe` 和 `ffprobe.exe`。

## 7. 数据、文件与错误处理

- 所有用户文件操作必须使用规范化后的明确路径；删除下载文件与仅删除记录必须是两个清晰选项。
- 不得递归删除工作区、下载目录、用户目录或由未校验变量拼出的路径。
- 重名策略必须严格执行 `rename / overwrite / skip`，不得静默覆盖用户文件。
- 运行时错误必须返回可操作的中文信息；底层错误可保留原因，但不得泄露密钥或完整认证头。
- Rust 运行时代码不得用 `unwrap()`、`expect()` 处理可恢复的网络、文件、数据库或用户输入错误。构建脚本和测试中的明确不变量除外。
- 不得为了绕过错误而吞掉失败结果。影响文件完整性、数据库迁移、安全校验和资源编译的错误必须使操作失败。

## 8. 代码与依赖注意事项

- 前端使用 React + TypeScript，桌面后端使用 Rust + Tauri，持久化使用 SQLite；避免引入重复框架或大型 UI 组件库。
- 新依赖必须有明确必要性，并评估体积、许可证、维护状态和供应链风险。可用标准库或现有依赖完成时不得新增依赖。
- 用户可见文案默认使用简体中文；代码标识、数据库字段和协议字段使用稳定英文。
- 保持事件驱动的增量任务更新，不得恢复高频全量轮询。
- 不得提交 `target/`、`dist/`、安装器、扩展 ZIP、媒体可执行文件、数据库、下载样本或临时测试文件。
- 不得覆盖或清理与当前任务无关的用户修改。提交应聚焦单一目的，并在提交前执行 `git diff --check`。
- 严禁编写超长代码或巨型单体函数/文件。必须自动遵循项目规范与最佳实践，在遇到复杂或长代码逻辑时，应主动将其拆解为独立、清晰的小型辅助函数、子模块或组件，确保单段代码职责单一，且易于测试和阅读。

## 9. 测试强约束

任何下载内核、数据库、桥接协议或安装器变更，必须提供与风险相称的测试。最低验证命令：

```powershell
pnpm run check
pnpm run build
cargo test --manifest-path src-tauri\Cargo.toml
pnpm run extension:build
```

- 多连接相关变更还必须运行本地 Range 服务，验证请求互不重叠、完整覆盖源文件、合并长度正确且 SHA-256 一致。
- 必须覆盖服务器忽略 Range、错误 Content-Range、未知长度、416、断网、重启续传、ETag 变化和损坏分片等回退路径。
- 数据库变更必须覆盖旧结构迁移与重启后的读取。
- UI 变更必须检查空任务、大量任务、长文件名、多选、排序、详情栏、浅色/深色及最小窗口。
- 扩展变更必须运行模拟 Chrome API 测试，至少覆盖配对、离线回退、接管开关和临时绕过。
- 不能用“能够编译”代替行为验证，也不能删除失败测试来让构建通过。

## 10. 发布强约束

- 发布版本必须同步更新根 `package.json`、`src-tauri/Cargo.toml`、`src-tauri/tauri.conf.json`、`extension/package.json` 和 `extension/manifest.json`。
- Windows 构建环境必须能找到 Windows SDK 的 `rc.exe`。`src-tauri/build.rs` 必须成功嵌入图标、Manifest 和版本资源；发布后需读取 EXE 文件属性确认 ProductName、ProductVersion 和 OriginalFilename。
- 发布命令：

```powershell
pnpm tauri build
```

- 发布后必须记录 NSIS、MSI、扩展 ZIP 的大小与 SHA-256，检查安装包文件列表，并启动 release EXE 验证 `/v1/health`。
- 不得发布带调试数据、演示任务、绝对开发路径、测试令牌或未校验媒体工具的构建。

## 11. 完成定义

只有在代码、迁移、测试、视觉检查、安装包检查和兼容性说明全部完成后，任务才能标记为完成。若某项无法验证，必须明确报告未验证内容和风险，不得用“应该可以”代替结果。
